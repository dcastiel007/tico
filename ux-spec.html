01 — User Personas

Who Uses t.Co?

Three primary personas represent the system's target audience. They are grounded in research and serve as the foundation for every design and specification decision in this document.

👩‍💼
Dana, 34
Product Manager · Tel Aviv
  • Travels 4–6×/year for business and leisure
  • Limited time — wants pre-organized, friction-free trips
  • Values personal recommendations over tourist traps
  • Advanced tech user; comfortable with voice interfaces
  • Loyal to specific hotel chains and airlines
Goal: "Plan me a trip that feels handcrafted, not algorithmic."
🧑‍🎓
Yoav, 26
Graduate Student · Haifa
  • Budget-conscious; prioritizes authentic local experiences
  • Strong interest in local culture and language immersion
  • Spontaneous — happy to change plans mid-trip
  • Relies on peer recommendations and social media
  • Comfortable with voice but prefers chat for privacy in public
Goal: "Show me what locals actually see and eat."

Racheli & Amir, 42/44
Family — Two kids (6 & 9) · Ra'anana
  • Travel 2×/year; highly sensitive to schedule disruptions

  • Strong preference for pre-confirmed, reliable bookings
Goal: "A trip everyone enjoys, with zero nasty surprises."
02 — User Journeys

Five Core Journeys

Each journey maps a complete usage scenario: steps, user emotion state, system actions, and active agents. J1, J2, and J4 are fully implemented in the exhibition demo. J3 and J5 are documented for architectural completeness and future development.

J1

Planning an Itinerary from Scratch

Persona: Dana Duration: 8–15 min Channel: voice + text Status: ✓ Implemented
AGT-01 Profiling AGT-02 Itinerary
01
App Open — First-Time Detection
System checks user_profile in PostgreSQL. No record found → new user flow initiated. Orchestrator dispatches AGT-01 with context: {session_id, timestamp, locale}.
PostgreSQL: SELECT user_profile WHERE user_id → NULL → new_user_flow()
Anticipation
02
Conversational Preference Profiling
AGT-01 conducts adaptive dialogue to extract 5 preference dimensions: destination intent, budget range, travel style, interests[], and constraints[]. Each response is parsed and stored incrementally. Profile confidence score computed after each turn.
AGT-01 → Claude API: extract_preferences(message) → ChromaDB: upsert preference_vector confidence_score ≥ 0.75 → trigger AGT-02
Engaged
03
Itinerary Generation — Real-Time
Orchestrator hands off preference_profile to AGT-02. AGT-02 calls Google Places for POIs, OpenAI for activity scoring, and assembles a day-by-day itinerary. Results stream to the frontend map in real time via WebSocket.
AGT-02 → Google Places API: nearby_search(destination, interests) AGT-02 → Claude API: rank_and_narrate(POIs, style) → WebSocket: stream itinerary_day[] to UI → PostgreSQL: INSERT itinerary(trip_id, days[])
Excitement
04
Iterative Refinement Loop
User requests changes via natural language: "Day 3 looks too packed", "Add a design museum." Orchestrator routes to AGT-02 with edit_intent. AGT-02 applies diff, increments itinerary.version, re-streams changed days only.
Orchestrator: parse_edit_intent(user_message) AGT-02: apply_diff(itinerary_id, edit_intent) → WebSocket: push updated_days[]
In control
05
Confirmation & Profile Save
User confirms. Orchestrator commits the itinerary, persists updated user_profile with inferred preference deltas. Optionally triggers email export via SendGrid.
PostgreSQL: UPDATE user_profile SET preference_vector PostgreSQL: UPDATE trip SET status='confirmed' SendGrid API: send_itinerary_email() [optional]
Satisfied
Edge Cases
Conflicting constraints (expensive destination + low budget) → AGT-01 surfaces trade-off dialogue; user selects priority dimension before AGT-02 proceeds.
Google Places API unavailable → fallback to Redis-cached POI dataset; UI surfaces transparent degradation notice.
Session interrupted mid-profiling → partial profile saved to Redis with 24h TTL; Orchestrator resumes on reconnect using session_id.
J2

Real-Time Language Bridging & Cultural Guidance

Persona: Yoav Duration: Ongoing (real-time) Channel: voice Status: ✓ Implemented
AGT-06 Language & Culture OpenAI Whisper STT DeepL Translate ElevenLabs TTS
01
Location-Aware Language Mode Activation
GPS coordinates passed to AGT-06. Agent resolves local language from coordinate-to-region mapping. If current language differs from user's native language → Language Mode activated proactively.
GPS → AGT-06: resolve_language(lat, lon) HERE Maps API: reverse_geocode() → country_code IF country_code ≠ user.native_locale → activate language_mode
Uncertain
02
Speech Capture & Transcription
User activates voice button. Audio recorded at 16kHz mono, buffered in 500ms chunks. Streamed to Whisper STT. Transcript returned with confidence score. Low-confidence segments flagged for re-prompt.
WebSpeech API: MediaRecorder(16kHz) → audio_chunk[] OpenAI Whisper API: transcribe(audio, language='auto') → {text, confidence} — if confidence < 0.7 → re-prompt
Focused
03
Translation + Cultural Annotation
Transcript passed to AGT-06. Agent calls DeepL for primary translation, then augments with Claude for cultural context, etiquette tips, and pronunciation hints. All returned as structured LanguageCard object.
DeepL API: translate(text, target_lang) AGT-06 → Claude API: annotate_cultural_context(text, location) → LanguageCard {translation, pronunciation, etiquette_tip, formality_level}
Empowered
04
Audio Output via TTS
Translation text passed to ElevenLabs with target language voice model. Audio streamed back in <1.2s. UI displays LanguageCard concurrently with audio playback. Session transcript logged to PostgreSQL.
ElevenLabs TTS API: synthesize(text, voice_id, language) → audio/mpeg stream → playback (target: <1.2s latency) PostgreSQL: INSERT translation_session(session_id, transcript[])
Confident
Edge Cases
Poor network → STT switches to on-device WebSpeech API (English only); translation cached phrases served from Redis offline store.
Local dialect / slang not in DeepL → Claude handles with explicit uncertainty flag in LanguageCard.confidence; user informed.
Unsupported language → AGT-06 returns nearest supported language with ISO code; graceful degradation, no silent failure.
J3

Complete End-to-End Journey

Persona: Racheli & Amir Duration: Before → During → After trip Status: ○ Conceptual
AGT-01AGT-02 AGT-03AGT-05 AGT-07AGT-08
S1
Stage 1 (Before) — Family-Aware Planning
AGT-01 profiles family composition including children's ages. AGT-02 filters POIs by family_friendly: true, applies activity duration limits, and schedules mandatory rest buffers. AGT-03 executes bookings with family room requirements and meal-time constraints.
AGT-01: extract_family_profile(children_ages[], dietary_restrictions[]) AGT-02: filter_poi(family_friendly=true, max_activity_duration=2h) AGT-03 → Amadeus API: book_flight(pax=4, family_seating=true) AGT-03 → Booking.com API: reserve_hotel(family_room, dates)
S2
Stage 2 (During) — Real-Time Schedule Recovery
AGT-05 subscribes to FlightRadar and OpenWeather webhooks. On disruption event: computes impact delta across downstream activities, generates 3 alternative schedules ranked by user preference alignment, presents to user for selection, then executes via AGT-03.
FlightRadar24 Webhook → AGT-05: disruption_event{flight_id, delay_minutes} AGT-05: compute_impact_delta(itinerary, delay_minutes) AGT-05 → Claude API: generate_alternatives(impacted_days, preferences) → user selects → AGT-03: rebook(new_schedule)
S3
Stage 3 (After) — Memory Curation & Learning
AGT-07 ingests photo metadata and location history to construct a narrative travel journal and curated album. AGT-08 collects explicit ratings and implicit signals (time-on-activity, skips), computes preference delta, and updates user_profile.preference_vector in ChromaDB.
AGT-07 → Google Photos API: fetch_trip_photos(trip_dates) AGT-07 → Claude Vision API: generate_narrative(photos, itinerary) AGT-08: compute_preference_delta(ratings[], behavioral_signals[]) → ChromaDB: update_vectors(user_id, preference_delta)
J4

Returning User with Travel History

Persona: Dana (Year 2) Duration: 3–5 min Channel: text Status: ✓ Implemented
AGT-01 AGT-02 AGT-11 Anticipation
01
Profile Retrieval & Personalized Greeting
On session start, Orchestrator queries user_profile from PostgreSQL. Existing profile with trip_count ≥ 1 → returning user flow. Last trip summary retrieved, injected into Orchestrator context window for informed greeting.
PostgreSQL: SELECT * FROM user_profiles WHERE user_id AND trip_count ≥ 1 PostgreSQL: SELECT summary FROM trips WHERE user_id ORDER BY end_date DESC LIMIT 1 Orchestrator: inject_context(last_trip_summary, preference_vector)
Recognised
02
Proactive Destination Suggestions via AGT-11
AGT-11 queries ChromaDB for similar-preference users' high-rated destinations not yet visited by Dana. Cross-referenced with seasonal pricing signals from Amadeus. Returns ranked suggestion list with reasoning.
AGT-11 → ChromaDB: similarity_search(preference_vector, exclude=visited_destinations[]) AGT-11 → Amadeus API: get_flight_inspiration(origin, budget_range) → ranked_suggestions[{destination, score, reasoning, est_price}]
Intrigued
03
Accelerated Itinerary Generation
Because preference_vector is pre-built, AGT-01 skips full profiling — only confirms any changes since last trip (<3 questions). AGT-02 generates an initial itinerary in ~40% of the time compared to a first-time user.
AGT-01: delta_profile_check(last_profile, current_session) → {changes_detected: bool} IF changes_detected → update_preference_vector() AGT-02: generate_itinerary(preference_vector, destination) — skips cold-start
Delighted
J5

In-Trip Schedule Recovery — Weather Disruption

Persona: Racheli & Amir (mid-trip) Trigger: external disruption event Status: ○ Conceptual
AGT-05 Schedule Optimizer AGT-03 Booking OpenWeather FlightRadar24
01
Disruption Detection — Proactive Monitoring
AGT-05 runs a background polling loop every 15 minutes against weather and flight APIs for all active trips. Detects that tomorrow's outdoor activities conflict with a high-probability rain forecast (>70%). Disruption event raised internally.
Cron: every 15min → AGT-05: poll_active_trips() OpenWeather API: forecast(lat, lon, days=3) → {precipitation_probability} IF precipitation_probability > 0.70 AND activity.outdoor → raise DisruptionEvent
Worried
02
Impact Assessment & Alternative Generation
AGT-05 scores each impacted activity by disruption severity. Calls Claude to generate 3 ranked alternative itineraries for the affected day, constrained by: open indoor venues, family-friendly, within budget delta ±15%.
AGT-05: score_disruption_impact(activities[], forecast) → impacted[] AGT-05 → Claude API: generate_alternatives(impacted_day, constraints={indoor:true, family:true, budget_delta:0.15}) Google Places API: search_indoor_alternatives(location, category[]) → alternative_itineraries[3] ranked by preference_score
Waiting
03
User Notification & Selection
Push notification sent via FCM with disruption summary and top alternative. In-app card presents all 3 options with estimated cost delta and duration. User selects. Orchestrator confirms selection and queues re-booking.
FCM: push_notification(user_id, {type:'disruption', summary, top_alternative}) UI: render AlternativeCard[3] with cost_delta, duration, preference_score → user.select(alternative_id) → Orchestrator: confirm_rebooking()
In control
04
Automated Re-booking Execution
AGT-03 executes cancellations within free-cancellation windows. Books new venues via respective APIs. Updates itinerary record in PostgreSQL. Sends confirmation digest to user. All within 90 seconds of user selection.
AGT-03 → Booking.com API: cancel_reservation(booking_ref) [if within free_cancel window] AGT-03 → OpenTable API: book_restaurant(venue_id, pax=4, datetime) PostgreSQL: UPDATE itinerary SET days[affected_day]=new_day WHERE trip_id SendGrid: send_rebooking_confirmation(user_email, updated_itinerary)
Relieved
Edge Cases
No indoor alternatives available at location → AGT-05 proposes full-day itinerary relocation to a nearby city with indoor attractions; cost delta flagged prominently.
Original booking outside free-cancellation window → AGT-03 presents penalty amount, asks explicit user confirmation before proceeding.
User ignores notification → system re-alerts 2h before the disrupted activity; if still ignored, falls back to original plan with weather advisory appended.

06 — Edge Case Matrix

Complete matrix of identified edge cases across all journeys. Priority: P0 = blocking, must be resolved before demo; P1 = degraded experience, acceptable; P2 = nice-to-have.

Whisper STT confidence < 0.7 J2 AGT-06 AGT-06 requests repeat with voice guidance ("please speak more slowly"); fallback to text input after 2nd failure P0
Conflicting profile constraints (low budget + luxury destination) J1, J4 AGT-01 Surfaces explicit trade-off dialogue; user selects priority dimension; AGT-02 proceeds with constrained generation P1
